Three Types of Sui Objects

üë§ Owned Objects

Belong to a specific address. Only the owner can modify or transfer. Enables parallel transaction processing.

Most Common

üåê Shared Objects

Accessible by anyone. Require consensus for modification. Used for DEXs, auctions, and multi-user applications.

Requires Consensus

üîí Immutable Objects

Cannot be modified after creation. Anyone can read. Perfect for configuration data and historical records.

Maximum Parallelism

Owned Object Example

public struct NFT has key { id: UID, name: String, image_url: Url, } public entry fun create_nft(name: String, ctx: &mut TxContext) { let nft = NFT { id: object::new(ctx), name, image_url: url::new_unsafe_from_bytes(b"https://example.com/nft.png"), }; // Transfer to transaction sender transfer::transfer(nft, tx_context::sender(ctx)); }
Use Case: Personal assets like NFTs, tokens, game items, or user-specific data.

Shared Object Example

public struct Auction has key { id: UID, highest_bid: u64, highest_bidder: Option<address>, } public entry fun create_auction(ctx: &mut TxContext) { let auction = Auction { id: object::new(ctx), highest_bid: 0, highest_bidder: option::none(), }; // Make accessible to everyone transfer::share_object(auction); }
Use Case: DEXs, auctions, DAOs, multiplayer games, or any multi-user state.

Immutable Object Example

public struct Config has key { id: UID, protocol_fee: u64, admin: address, } public entry fun publish_config(fee: u64, ctx: &mut TxContext) { let config = Config { id: object::new(ctx), protocol_fee: fee, admin: tx_context::sender(ctx), }; // Make permanently immutable transfer::freeze_object(config); }
Use Case: Protocol configurations, reference data, constants, or historical records.

Sui vs Ethereum: Object Models

Aspect Sui Objects Ethereum Accounts
State Model Object-centric (each object independent) Account-centric (global state tree)
Parallelism ‚úÖ High (owned objects don't conflict) ‚ùå Low (shared global state)
Ownership Explicit (built into the model) Implicit (managed by contracts)
Transaction Conflicts Rare (only with shared objects) Common (popular contracts bottleneck)
Gas Efficiency Pay per object accessed Pay per computation
Composability Object wrapping, dynamic fields Contract calls
Best For High-throughput, asset-focused dApps Complex smart contract interactions
Key Insight: Sui's object model enables massive parallelization because transactions touching different objects can execute simultaneously. This is like having multiple cashiers in a store vs. one global queue in Ethereum.
